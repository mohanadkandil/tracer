Rust workers, in plain English

A learning doc for someone new to Rust. What a worker is, why we'd port one layer (not the whole stack), and what the actual code looks like.

What is a "worker" in our system

The Python scan service today does two jobs in one process:

1
Gateway — handles HTTP requests from the UI. Low traffic. Humans clicking buttons.
2
Worker — pulls file events off a queue, fetches bytes, runs the model, writes findings. This is the hot loop. Runs millions of times a day at production scale.

The worker is the part that scales. The gateway runs in 1 pod forever. The worker runs in 10, then 100, then 1000 pods as Bosch grows.

key idea

"Rust workers" = take the worker job only and rewrite it in Rust. Gateway, frontend, training scripts, everything humans touch stays Python. The model itself stays in PyTorch / ONNX.

Throughput barely moves. The model takes 80 ms no matter who calls it. The win is density — fitting 20× more workers on the same hardware, and starting them in 20 ms instead of 3 s when load spikes.

You don't write Rust for speed.
You write Rust for the unit economics at the 100th pod.

What stays Python

FastAPI gateway
Low-rps, human-interactive, latency is human-perceptible. Python is fine forever.
Admin / frontend BFF
Hit a few times per second per user. No win.
Training (train/)
PyTorch lives in Python. The whole ML world does.
Data generation (data_gen/)
One-shot scripts. Run by humans. Dev tooling.
Mosaic embedder + DSAR reasoning
Python orchestration around a model. ONNX call dominates anyway.

The worker code, narrated

Here is the entire worker loop. ~80 lines. Below it, each block explained in plain English.

// worker.rs — pulls files from a queue, runs the PII pipeline,
// writes findings to Postgres. Stateless. Crash-safe (queue retries).

use anyhow::Result;
use sha2::{Digest, Sha256};
use sqlx::PgPool;
use tokio::sync::Semaphore;
use std::sync::Arc;

#[derive(Debug, serde::Deserialize)]
struct FileEvent {
    id: String,
    source: String,
    path: String,
    fetch_url: Option<String>,
    mime: String,
    owner: Option<String>,
}

#[tokio::main]
async fn main() -> Result<()> {
    let db = PgPool::connect(&std::env::var("DATABASE_URL")?).await?;
    let queue = redis::Client::open(std::env::var("REDIS_URL")?)?
        .get_multiplexed_async_connection().await?;
    let model = onnx::Session::builder()?.load("/models/gliner-bosch.onnx")?;
    let concurrency = Arc::new(Semaphore::new(64));

    loop {
        let event: FileEvent = queue.xread_one("scan-jobs").await?;
        let permit = concurrency.clone().acquire_owned().await?;
        let db = db.clone();
        let model = model.clone();
        tokio::spawn(async move {
            let _permit = permit; // drops on scope exit
            if let Err(e) = process_one(event, &db, &model).await {
                tracing::error!(?e, "worker failed");
            }
        });
    }
}

async fn process_one(ev: FileEvent, db: &PgPool, model: &onnx::Session) -> Result<()> {
    let bytes = fetch_bytes(&ev).await?;
    let hash = format!("{:x}", Sha256::digest(&bytes));

    // 1. Dedup gate
    if let Some(cached) = lookup_cached(db, &hash).await? {
        write_findings(db, &ev, &cached).await?;
        return Ok(());
    }

    // 2. Text extraction
    let text = match ev.mime.as_str() {
        "application/pdf" => pdf::extract(&bytes)?,
        "application/vnd.openxmlformats-officedocument.wordprocessingml.document"
            => docx::extract(&bytes)?,
        _ => String::from_utf8_lossy(&bytes).into_owned(),
    };

    // 3. Tier 1 + Tier 2 (regex + ONNX model)
    let t1_spans = presidio_rs::detect(&text);
    let t2_spans = model.predict_entities(&text, &ENTITY_LABELS, 0.5)?;
    let spans = merge_overlapping([t1_spans, t2_spans].concat());

    // 4. Persist + cache
    cache_spans(db, &hash, &spans).await?;
    write_findings(db, &ev, &spans).await?;
    Ok(())
}

Block-by-block

A
#[tokio::main] — Tokio is Rust's async runtime, like Python's asyncio. The annotation means "run main as an async task on the Tokio scheduler." Lets us write .await everywhere.
B
PgPool::connect — a connection pool to Postgres. Same idea as Python's SQLAlchemy engine. sqlx is the popular async Postgres driver in Rust.
C
Semaphore::new(64) — caps concurrency at 64 files in-flight per worker process. Stops us from OOMing if the queue dumps 10K events at once. This is the magic part — one Rust process holding 64 fetches in flight uses ~50 MB. The Python equivalent uses ~500 MB and starts swapping at 200.
D
loop { … queue.xread_one … tokio::spawn(…) } — the worker loop. Pull one event, acquire a permit, spawn an async task to process it, immediately go back for the next event. Tokio's scheduler keeps thousands of these in-flight without ever blocking the main loop.
E
Sha256::digest(&bytes) — the dedup gate. SHA-256 in Rust is ~5× faster than Python's hashlib because there is no GIL between you and the hardware instruction. But honestly, hashing a 100 KB file in either language is sub-millisecond — this is not where Rust shines.
F
model.predict_entities — the ONNX session call. This is the same model file Python would call. We exported the fine-tuned GLiNER to ONNX once; both languages run it identically. ~80 ms / doc. Language host irrelevant here.
G
match ev.mime.as_str() — Rust's match is like Python's match but exhaustive. The compiler will refuse to build if you add a new MIME type and forget to handle it. This is the "boring safety" Rust gives you: a forgotten branch is a compile error, not a 2 AM page.
H
Result<()> on every function — Rust forces you to handle every possible failure. You cannot accidentally swallow an exception. ? propagates errors up. The runtime panic surface is tiny compared to Python.

Cost model — why bother

This is the slide for the pitch deck. Numbers are projected from public benchmarks of comparable workloads.

ScalePython workersRust workersSavings
Hackathon demo$0$00%
Bosch v1 (10K drives, 10M docs/day)$8K / mo$2.5K / mo~3.2×
Bosch v2 (300K drives, 50M docs/day)$45K / mo$12K / mo~3.7×
DAX-40 (10× Bosch)$450K / mo$110K / mo~4.1×

The savings come from memory density + autoscale efficiency, not raw single-doc speed. A k8s pod that needs 1 GB RAM costs more than one that needs 64 MB, even if both do the same work.

When to actually port

01
today   Python workers. Ship the hackathon demo. Run it on a laptop. Rust would be premature optimization.
02
pilot   Still Python. <5 M docs / day. Saving $2K / mo is not worth a rewrite.
03
v1 prod   Rust worker for the hot loop only. ONNX session inside. Keeps Python everywhere a human looks.
04
scale-out   Rust workers are required for unit economics. Without them, the model server bill grows linearly with customer count.

What to study while you wait

Three things give you 80% of what you need to read the worker above:

The Book
The official Rust Book — chapters 1–10, plus 16 (concurrency) and 19 (advanced types). doc.rust-lang.org/book
Tokio tutorial
Async in Rust looks alien at first because of lifetimes. Tokio's official tutorial walks you through the model. tokio.rs/tokio/tutorial
Rust by Example
Concrete code you can run. Faster than the Book for some learners. doc.rust-lang.org/rust-by-example

Concepts to grok in this order

1
Ownership & borrowing — the thing that makes Rust feel weird. Every value has one owner. Borrowing rules prevent data races at compile time. The mental model takes a week. After that everything clicks.
2
Result<T, E> and ? — error handling without exceptions. The ? operator unwraps a Result or returns the error up the stack.
3
Traits — like Python's Protocols or Go's interfaces. Behaviour you implement for a type.
4
async / await — looks like Python's async, runs differently. Futures are lazy; nothing happens until you .await or hand it to a runtime.
5
Tokio — the runtime that actually runs your futures. Knows about threads, IO, timers.

Honest TL;DR

verdict

Don't port now. Python is fine for the demo, fine for the pilot, fine until you have actual paying customers. When the cloud bill starts hurting, port just the worker loop. Everything else stays as is. The model file is portable; that's the whole trick.
